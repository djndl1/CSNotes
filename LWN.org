- [[https://lwn.net/Articles/4974/][Read-copy-update]]: An explanation of RCU (Read-copy-update), a synchronization mechanism optimized for frequently-read, rarely-modified data. RCU allows lock-free reads by having writers make copies, modify them, atomically update pointers, then defer freeing old copies until no references remain. The kernel tracks "quiescent states" (times when CPUs are idle or in user space) to determine when it's safe to reclaim memory. Addresses concerns about the preemptive kernel requiring additional handling via rcu_read_lock() or atomic counters.

- [[https://lwn.net/Articles/5500/][Thrashing the interrupt code]]: Describes the removal of cli()/sti() and the global_irq_lock from Linux 2.5.28. Ingo Molnar's patch eliminated the global interrupt disabling mechanism that disabled interrupts on all processors, replacing it with per-processor interrupt control. The patch also merged three kernel counters (hard IRQ, soft IRQ/bh, and preemption counters) into a single per-process preempt_count. Drivers that relied on cli()/sti() need to be converted to use proper spin_lock_irq() for protection.

- [[https://lwn.net/Articles/6403/][The asynchronous I/O core]]: Describes Ben LaHaise's AIO (asynchronous I/O) subsystem patch for Linux. The patch provides the core mechanism for submitting, tracking, and cancelling asynchronous I/O operations via new syscalls io_submit() and io_getevents(). File operations (aio_read, aio_write, aio_fsync) must be implemented in file_operations to support AIO. The article notes that actual execution of AIO operations was still missing - devices and filesystems needed to implement the new operations. The patch included support for mapping completion buffers into user space to avoid syscalls, and handling for cancellation when applications exit.

- [[https://lwn.net/Articles/7577/][Making Linux safe for pthreads]]: Describes Ingo Molnar and Rusty Russell's work to improve Linux thread support for pthreads. Key contributions: (1) Futex (fast user-space mutex) - allows pthread mutex implementation that only syscalls on contention; (2) TLS (thread-local storage) patch using x86 GDT segments with set_thread_area() syscall; (3) New clone() flags: CLONE_SETTID (writes thread PID back to parent), CLONE_SETTLS (sets up TLS), CLONE_RELEASE_VM (flag for safe thread stack freeing), CLONE_DETACHED (no SIGCHLD on exit). Each CPU got its own GDT, making context switches faster. The goal was efficient thread creation/destruction with minimal syscalls.

- [[https://lwn.net/Articles/6969/][Large page support in the Linux kernel]]: Discusses Linux kernel support for large pages (up to megabytes) to reduce page table overhead and improve TLB (translation lookaside buffer) efficiency. Linux traditionally uses 4KB/8KB pages; large pages can reduce memory overhead (1024 PTEs for 4MB) and improve performance by extending TLB reach. Rohit Seth's patch introduced get_large_pages()/share_large_pages() syscalls but handled large pages outside normal VM by reserving 25% of memory at boot. Linus noted this was really a "TLB interface" patch. Future wishes included: large page file I/O, mmap flags for large pages, and automatic large page allocation when applications request enough memory. The kernel already uses large pages for mapping the kernel image itself.

- [[https://lwn.net/Articles/8228/][How random is random enough?]]: Oliver Xymoron posted patches analyzing /dev/random entropy collection, claiming the kernel overestimates entropy in interrupt timing (resolution not as good as assumed, and timing can be manipulated by hostile users), and that entropy transfer between primary/secondary pools was broken. Ted Ts'o responded explaining the design. The fix would reduce available entropy, making /dev/random harder to use. Linus objected, suggesting if /dev/random can't provide useful randomness, it might be better to remove it entirely. The discussion showed tension between "proper" entropy accounting and practical usability.

- [[https://lwn.net/Articles/8720/][The scheduler and hyperthreading]]: Describes Ingo Molnar's scheduler patch for hyperthreading optimization. Hyperthreading allows a single CPU to behave as two virtual processors; when one stalls, the other runs. However, a naive SMP scheduler doesn't understand the difference between virtual processors on the same physical CPU (which share caches) versus separate physical CPUs. The problem: with 2 physical CPUs × 2 virtual processors each, two processes might both run on the same physical CPU when migrating one to the other physical CPU would perform better. Solution: change run queues from per-processor to per-physical-CPU, allowing virtual processors to share a queue. Smart idle detection (all virtual processors must be idle), plus cheaper task migration between sibling virtual processors. Result: 10%+ performance improvement.

- [[https://lwn.net/Articles/9620/][Multics security, thirty years later]]: Summary of Karger and Schell's paper comparing Multics security (1974) to modern systems. They found modern "security enhanced" systems lack even basic Multics security techniques. Key quote: SELinux module (1.7MB) is 2.5x larger than Multics' entire 1973 kernel (628KB), questioning whether complexity helps security. They advocate verifiable "security kernels" and note PL/I's explicit string length declarations prevented buffer overflows (unlike C). Prediction: without major improvements, either horrific cyber disasters or effective security technology delivery will occur in 30 years. Also mentions KeyKOS and EROS as successors to Multics security ideas.

- [[https://lwn.net/Articles/10106/][A new way to sleep?]]: Describes Andrew Morton's new safe sleep API in 2.5.35-mm1. Traditional sleep patterns like "while (condition) interruptible_sleep_on(queue)" have race conditions where wakeups between checking and sleeping can be missed. The new API uses prepare_to_wait() and finish_wait() to safely bracket the sleep, cleaning up the error-prone pattern of manually managing wait queues, task states, and the scheduler. Example: DECLARE_WAKE_QUEUE; prepare_to_wait(&wait_queue, &entry, TASK_INTERRUPTIBLE); if (condition_not_met) schedule(); finish_wait(&wait_queue, &entry);

- [[https://lwn.net/Articles/10238/][Solving the process ID allocation problem]]: Describes Ingo Molnar's new PID allocator to fix get_pid()'s quadratic behavior with large numbers of processes (100,000+). Uses bitmap pages with fast bit finding - claims <10us to allocate PID even with 999,999 PIDs allocated. Builds on William Lee Irwin's "idtag" hash table infrastructure for session/process group IDs. Linus preferred a simpler fix: just make max PID very large so the quadratic code path never runs. Ingo argued a large PID space can be exceeded; better to fix the algorithm. Linus put a simpler large-PID-space patch in his tree.

- [[https://lwn.net/Articles/11351/][The end of task queues]]: Describes Ingo Molnar's workqueue patch replacing task queues in Linux. Old mechanisms: bottom halves (32 max, compile-time numbers, couldn't sleep), then task queues (linked lists of functions, run from scheduler/timer interrupt). Problems: not transparent, still bottom halves, keventd single thread meant one slow task blocked all others. New workqueue mechanism: hidden internals, per-CPU worker threads (so one subsystem's slow task doesn't block others), executes in process context (can sleep), has default workqueue for simple tasks. Task queues removed in 2.5.40, workqueue patch pending merge.

- [[https://lwn.net/Articles/14168/][sys_epoll - making poll fast]: Describes Davide Libenzi's epoll syscall for scalable I/O event notification. Unlike select()/poll() which scan all file descriptors each call and don't scale with thousands of fds, epoll uses persistent kernel data structures. Three syscalls: epoll_create(maxfds) creates an epoll fd, epoll_ctl(epfd, op, fd, events) adds/changes/removes fds to watch, epoll_wait(epfd, events, timeout) waits for events. Uses callback-based notification - adds callbacks to struct file, avoiding scanning. Scales in constant time to huge numbers of fds. Initially for pipes/sockets, extensible to other I/O. Similar to NT's IO Completion Ports. Benchmarked to handle 60K sockets on 1GHz PIII.

- [[https://lwn.net/Articles/14776/][Initramfs arrives]]: Describes the initramfs feature merged in 2.5.46 - a cpio archive attached to the kernel image, unpacked into RAM at boot and used as initial root filesystem. Advantages: smaller kernel code, easier early boot customization without patching kernel, C library and memory protection for initialization code, cleaner interface via syscalls instead of in-kernel "magic". Plan: create klibc (small C library), kinit app, move initrd handling and root mounting to user space. ramfs (not a block device) used instead of traditional ramdisk - simpler and more space-efficient since no duplication between RD block device and dentry/inode caches.

- [[https://lwn.net/Articles/18411/][How to speed up system calls]: Describes the vsyscall page optimization merged in 2.5.53 to speed up slow int $0x80 syscalls on Pentium 4. Problem: int $0x80 is very slow on P4; solution: use sysenter instruction instead (much faster on modern CPUs). Solution: kernel maps a read-only "vsyscall" page at top of each process's address space containing optimal syscall code for the current processor. glibc jumps to this page instead of using int $0x80. Challenge: sysenter only supports 5 arguments, so Linus devised a tricky workaround. Result: gettimeofday() runs in half the time on P4, 1.2x faster on P3. Note: sysenter support varies by CPU (Athlon/Duron yes, K6 no).

- [[https://lwn.net/Articles/19398/][Two views of freedom and software]: Covers Richard Stallman'slinux-kernel posts on binary-only kernel modules and software freedom. Two philosophies: Stallman's view that proprietary software always harms freedom by denying rights to study/change/redistribute; versus view that freedom means authors choose licensing and users choose software. Andre Hedrick briefly withdrew from Linux development over this issue. Resulting consensus: binary-only modules acceptable if they stick to exported API; kernel header files (including inline functions) are part of the exported API. This clarified what proprietary module vendors can and cannot do.

- [[https://lwn.net/Articles/21274/][Anticipatory I/O scheduling]: Describes Nick Piggin's anticipatory I/O scheduler in 2.5.59-mm5. Problem: disk seeks are slow, prioritizing reads (to keep responsiveness) conflicts with grouping writes (which needs backlog). 2.4 mixed reads with writes, causing slow response during write activity. Even with reads-to-head fix, creates "seek storms" - constant seeking between reads and writes. Solution: after handling a read, wait briefly (millisecond timer) for another "close" request before resuming writes. If close requests keep coming, keep servicing them; otherwise resume normal dispatch. Nearly 6x performance improvement in testing. The anticipatory scheduler can be added to any disk scheduler.

- [[https://lwn.net/Articles/21379/][Fast reader/writer locks]: Describes Stephen Hemminger's fast reader/writer locks (frlocks, later renamed seqlock) for solving gettimeofday() performance and denial-of-service issues. Problem: reader/writer spinlocks allowed frequent gettimeofday() calls to block timer tick updates. Solution: frlocks use "pre" and "post" sequence numbers - writers increment pre, modify data, increment post. Readers check that post sequence matches remembered pre sequence (meaning no writer modified data during read). Neither blocks; readers just retry if collision detected. 18% improvement in gettimeofday() speed. Not suitable for lengthy calculations or operations with side effects.

- [[https://lwn.net/Articles/23037/][Synchronous signal handling]: Describes Linus's sigfd() syscall prototype for signal handling. sigfd(mask) returns a file descriptor that reports signals matching the mask; reading returns signal structures one at a time. Advantages: orderly signal handling without races, can use select/poll with signals, can pass fd to another process. Linus noted it could extend to timers and async I/O but didn't want to over-extend the interface. Discussion also covered Davide Libenzi's virtual filesystem patch to eliminate duplicated filesystem/inode boilerplate code across kernel subsystems.

- [[https://lwn.net/Articles/23687/][Full disclosure and the banking industry]]: Commentary on security research by Ross Anderson's team revealing ATM PIN vulnerabilities. Researchers Mike Bond and Piotr Zielinski found a flaw in the decimalization table used in PIN generation that allowed extracting PINs in ~15 guesses instead of ~5000. Citibank sought gag order to suppress disclosure; paper spread widely once revealed. Discusses the broader issue of full disclosure vs. security through obscurity, citing Matt Blaze's master-key lock vulnerability. Emphasizes that banking industry needs external review like software industry. Notes burden of proof differences between US (bank must prove customer fraud) vs UK (customer must prove innocence).

Issue: request queue structures consume ~150MB for 4000 disks; mempool mechanism used to dynamically allocate request structures. Roman Zippel opposed char device changes, preferring old char device hashing. Discussion on whether dev_t should be 32 or 64 bits. Solution uses mempool for dynamic request structure allocation.

Explains why X succeeded despite criticisms: hid hardware details, network transparency, allowed policy experimentation, extensible protocol, worked and was free. Despite being called "vacuumware" and criticized for being bloated, no viable competitor emerged in 15+ years. - [[https://lwn.net/Articles/28324/][The ongoing device number debate]: Continues dev_t expansion discussion. The expansion itself is uncontroversial; the real debate is whether to implement fully dynamic device number space. Roman Zippel advocated for full dynamic approach, but others want to solve immediate problems first (like SCSI disk number shortage). devfs is a solution but disliked by many; hotplug pushes the problem to user space but isn't ready. Decision: 2.6 will likely get expanded dev_t, with fully dynamic approach deferred to 2.7.

- [[https://lwn.net/Articles/28897/][Managing dynamic device naming]: Describes Greg Kroah-Hartman's udev project, pushing device naming entirely to user space. Three components: namedev (naming policy), libsysfs (common API for device info from sysfs), udev (responds to hotplug events). Current version 0.1 is minimal proof of concept; issues to resolve: race conditions in event management, multiple programs wanting hotplug events, per-site naming policy chaos. Linux Standard Base designates LANANA devices.txt as authority for standard names. 2.6 will have larger dev_t and udev hooks; full solution likely for 2.7.

- [[https://lwn.net/Articles/29906/][Novell and Linux]: 2003 interview with Novell's Kristopher Magnusson after SUSE acquisition. Novell making NetWare 7 run on Linux kernel ("dual-kernel" strategy). - [[https://lwn.net/Articles/33812/][strlcpy()]: Discusses OpenBSD's strlcpy() function added to Linux kernel. Problem: classic C string functions strcpy() are unsafe, but strncpy() is poorly suited - wastes time zero-filling destination, and doesn't guarantee NUL-termination if truncated. strlcpy() signature: size_t strlcpy(char *dest, const char *src, size_t size) - copies src to dest, always NUL-terminates, returns source length so caller can detect truncation. Linus merged it into kernel after getting fed up with buffer overflows. Also covers OpenBSD's related strlcat(). Discussion about whether glibc should include it (Ulrich Drepper opposed). Multiple public domain implementations posted for handling overlapping memory regions.

- [[https://lwn.net/Articles/31875/][What is "unauthorized access"?]: Analyzes legal definitions of "unauthorized access" in US computer crime law. Problem: laws are vague about what constitutes accessing a computer and what makes it unauthorized. Robert Morris worm case - did he "access" all infected systems or just launch point? Paper by Orin S. Kerr proposes: "access" should be broad (any successful interaction with computer), but "unauthorized" should be narrow (only circumventing code-based access controls). Breaching contract (ToS, spam, deep linking) should be civil matter, not criminal. Anti-circumvention makes sense for access controls as "lock on the door." Concerns: overly broad interpretations could criminalize everyday net behavior.

- [[https://lwn.net/Articles/30653/][The Linux kernel and digital rights management]: Discusses Linus's statement that DRM is "perfectly ok with Linux." Hardware can refuse to boot kernels not signed with a particular private key, restricting what runs on the system. GPL argument: private keys aren't covered by GPL source code definition, so distributing signed binaries without keys is allowed. DRM techniques could also enable higher security (signed executables). Linus doesn't want to discriminate against DRM users per free software principles. Ultimately marketplace must decide; GPL can't prevent hardware-locked systems.

- [[https://lwn.net/Articles/30773/][Suspending the system]: Describes software suspend patch merged in 2.5.18. Process: send freeze signals to processes (go to STOPPED state), free memory, flush disk writes, suspend devices, write page directory to swap partition, power down. Resume detects image and restores state. Problems: no SMP, no high memory, x86 only, needs adequate swap partition. Issues: swap file with journaling filesystems causes corruption (journal replay conflicts with restored state). Patrick Mochel working on general power management framework that can hide suspend/resume behind function pointers, adapting to hardware capabilities (ACPI vs software suspend).

- [[https://lwn.net/Articles/31462/][The right way to yield]: OpenOffice performance regression from sched_yield() semantics change in 2.5. Old behavior: moved process to end of run queue. New behavior: moves to "expired" queue, cancelling time slice - process must wait for all others to exhaust slices. OpenOffice used busy-wait loops with sched_yield(), causing poor responsiveness with new semantics. Consensus: apps using sched_yield() for busy-wait are "broken" and should be fixed. OpenOffice fix already committed. Discussion: sched_yield() now only useful when truly giving up CPU with no time slice; use conditional variables for waiting.

- [[https://lwn.net/Articles/21861/][The MS-SQL worm: lessons for free software]: Analysis of the MS-SQL (Slammer) worm and lessons for free software. Worm infected ~100,000 hosts within minutes, no time for admins to react. Monoculture concerns: eliminating monocultures won't make net safe; any widespread vulnerability in free software could be used similarly. Free software advantages: security updates are small fixes, easy to install; no bundled subsystems users don't know about (SQL Server hidden in other apps); databases like PostgreSQL/MySQL don't expose to net by default. Suggestions: egress filtering, connection rate limiting at border routers. SQL Slammer specifically spread fast due to UDP-based probing (no TCP handshake delay). Rate limiting new IP connections (e.g., 10 new/second) could slow worm spread while not affecting normal use.

- [[https://lwn.net/Articles/40010/][RFCs - insufficiently free?]: Discusses Debian bug 92810 about RFC licensing. RFCs are core Internet standards documents, but carry a copyright notice restricting modifications - only allowed for developing Internet standards, not creating derived works. This violates Debian Free Software Guidelines (DFSG). Debian removed RFCs from main distribution to non-free. Internet Society's intent: RFCs should be immutable once issued (changes come via new RFCs with new numbers). Critics argue license prevents forking if ISOC goes bad. Discussion: standards vs documents, trademark vs copyright, whether ISOC's restrictions are reasonable. Solutions proposed: trademark model (can use name if unmodified), or require clear labeling of derived versions. Debian serves as "conscience" for licensing issues even when few users affected.

- [[https://lwn.net/Articles/39925/][64GB on 32-bit systems]: Discusses memory limitations on 32-bit x86 systems. Linux historically limited to 1GB kernel address space due to 3GB/1GB virtual address split. High memory support allows >1GB but kernel data structures still limited to low memory. Problem: struct page for each physical page consumes 40 bytes/page; 64GB RAM needs 640MB just for memory map, exceeding 1GB low memory. Ingo Molnar's patch: separate page tables for user/kernel, eliminating split - quadruples low memory to 4GB. Tradeoff: TLB flush required when switching page tables, causing significant slowdown. Not likely in mainline before 2.7; may appear in vendor distributions. Alternative: page clustering, or simply use 64-bit hardware.

- [[https://lwn.net/Articles/44266/][Too many threads?]: Discusses proliferation of kernel threads in 2.6. Problem: firmware loading via default workqueue had latency issues; solution created dedicated workqueue but added more kernel threads. Author's dual-Pentium system shows 21 kernel threads (migration, ksoftirqd, events, kblockd, khubd, kirqd, pdflush, kswapd0, aio, scsi_eh, ahc_dv, kseriod, kjournald). Kernel threads consume memory and clutter ps listings. Proposed rework: turn default workqueue into thread pool that creates threads on demand and trims via timeout. Many workqueue users don't need per-CPU threads - could specify single thread. Changes not difficult but questionable if appropriate at this development stage.

- [[https://lwn.net/Articles/46678/][dev_t expansion status]: Status update on 64-bit dev_t expansion. Had been stalled waiting for Alexander Viro's patches, now appearing in 2.6.0-test4-mm2. Patches include: cleanup of kdev_t usage (replaced with proper types), JFFS filesystem changes (explicit 16-bit device field for on-media structures), and new iminor()/imajor() helper functions to access minor/major numbers without knowing internal storage. Future patches will change inode structure to use dev_t instead of kdev_t for i_rdev. Important feature for 2.6 to support larger device numbers.

- [[https://lwn.net/Articles/50703/][dev_t expands at last]: dev_t expansion merged into 2.6.0-test6. Final format: 32 bits total - 12 for major, 20 for minor (vs earlier proposal of 32/32). Encoding designed to maintain backward compatibility: old 16-bit device numbers remain valid in new format. User-space encoding differs from internal kernel encoding to preserve compatibility. Problem: some ioctl() calls and filesystems pass 16-bit values directly - these return EOVERFLOW error if value doesn't fit. Discussion: EOVERFLOW vs filling in illegal device number (deferring error to actual use) - similar to readdir32 issues. Linus merged the changes; expect some surprises as remaining issues surface.

- [[https://lwn.net/Articles/53806/][Two views of the Unix philosophy]: Comparative review of two books on Unix philosophy. "The Art of Unix Programming" by Eric Raymond (available online under Creative Commons) vs "Linux and the Unix Philosophy" by Mike Gancarz (1994 book retrofitted with Linux references). Raymond's book is more comprehensive, covers scripting, networks, modern languages; Gancarz's book shows its age (advice like fitting function params on one line). Key philosophy contrasts from the books: Raymond: write big programs only when nothing else will do, prototype before polish, choose portability over efficiency, design programs to connect with others, store data in flat text files, silence is golden, separate policy from mechanism. Gancarz: small is beautiful, make each program do one thing well, programmer time is expensive, use shell scripts for leverage. Both books neglect security. Recommended: TAOUP over Gancarz's book.

- [[https://lwn.net/Articles/56739/][Loading NDIS drivers into the kernel]: Describes ndiswrapper, a GPL'd Linux module that loads Windows NDIS drivers. Works by registering misc device with ioctl() to accept driver from user space, performing relocation, then providing two glue layers: net_device operations for networking subsystem, and emulation of Windows API calls (interrupt allocation, I/O memory, packet feeding). NDIS API is well documented by Microsoft. Criticisms: wrapper approach is suboptimal (no performance/reliability improvements), requires proprietary Windows code in kernel, may discourage vendors from native Linux drivers, only works on x86. Alternative to proprietary Linuxant DriverLoader. Could be useful for reverse-engineering unsupported hardware.

- [[https://lwn.net/Articles/56923/][The future of the Linux filesystem]: Discusses Linux filesystem competition with Microsoft's WinFS (SQL-based filesystem in Longhorn). Two Linux approaches: (1) ReiserFS - Hans Reiser's vision of no structure imposed by filesystem, free-form search queries, everything including databases in filesystem. Reiser4 adds transactions, better small file performance, plugin architecture. (2) GNOME Storage - entirely user-space, built on gnome-vfs, uses PostgreSQL underneath, aims to replace traditional filesystem with document store. Neither proposes putting SQL in kernel. By Longhorn's expected release, Linux may have comparable alternatives.

- [[https://lwn.net/Articles/57800/][On the proper use of vmalloc()]: Discusses proper vmalloc() usage. Linus stated vmalloc() should have no valid new uses - only legitimate for modules and a few legacy cases (like file descriptor array). Every vmalloc() call requires page table tweaking and TLB flushes, making it slow. vmalloc() space is outside regular kernel range, requires extra TLB slots, and virtual space is limited on many architectures. For large allocations, use seq_file interface instead. Example: /proc/interrupts buffer allocation on large SMP systems failed with kmalloc() - but Linus preferred seq_file solution over vmalloc().

- [[https://lwn.net/Articles/64327/][Use of patented code in the kernel]: Linus clarified kernel policy on patents: don't submit code implementing known patented algorithms, find unencumbered alternatives instead. Known problems should be excluded, but developers shouldn't go looking for patent issues. Proposal to add CONFIG options for patented code rejected - would create partially proprietary kernel. Some patented code already exists: SB-Live mixer code with undisclosed license, FTL patent for memory card driver (only licensed for PCMCIA). Future difficult choices likely as more patent-encumbered code is discovered.

- [[https://lwn.net/Articles/65178/][Kernel threads made easy]: Rusty Russell's new kthread API simplifies kernel thread creation for hotplug CPU work. New functions: kthread_create() (creates but doesn't start), kthread_run() (create and start), kthread_stop() (stops thread via signal), kthread_bind() (pin to CPU). Thread function signature: int thread_function(void *data). Thread checks signals and returns when asked to stop. Patches convert existing kernel thread users to new infrastructure.

- [[https://lwn.net/Articles/65195/][The future of device numbers]: Greg KH explained udev vs devfs to reduce devfs user complaints. Linus clarified device numbers are "random cookies" not stable identifiers - attempts to make them stable just encourage buggy code. Hotplug devices (USB, iSCSI) already have unstable numbers. udev moves naming policy to user space where it belongs. Proposal: random device numbers during development to flush out assumptions. devfs deprecated but will be supported through 2.6 lifetime.

- [[https://lwn.net/Articles/66091/][User-space device drivers]: Gelato project (IA-64) work enabling device drivers in user space. Drivers are 50% of kernel code but cause 85% of bugs. User-space drivers allow stable ABI, any language (even Python), no licensing issues. Required pieces: I/O port access, mmap /dev/mem, PCI config space via ioctl(). Missing: interrupt handling - Gelato patch adds /proc/irq/N/irq interface; reading blocks until interrupt, then returns to user space. DMA: new syscalls usr_pci_open(), usr_pci_map() with USR_ALLOC_CONSISTENT/USR_MAP/USR_UNMAP. User-space block drivers register with kernel via special files. Concerns: some see this as microkernel-like; could legitimize binary-only drivers. Linux.Conf.Au presentation.

- [[https://lwn.net/Articles/66829/][FUSE - implementing filesystems in user space]: FUSE (three-part system): kernel module hooks into VFS, special device for user-space comms, protocol translation; userspace library translates requests to function calls (open, read, write, rename, symlink etc); user code fills fuse_operations struct. AVFS uses FUSE for tar/zip filesystems. Could mirror web sites, access object databases, experiment with "everything is a file." FUSE 1.1 release candidate announced. Security: only mount requires root, fusermount allows non-root users. KDE KIO slaves via FUSE gateway exists.

- [[https://lwn.net/Articles/67194/][User-space filesystems and security]: Security discussion around FUSE. fusermount handles secure mounting for non-root users. Concern about running filesystems from untrusted sources. Generally safe if only root can mount. See also LUFS, AVFS, userfs projects.

- [[https://lwn.net/Articles/68104/][NFS in user space?]: NFS server traditionally ran in user space while client was in kernel. Using user-space filesystems like FUSE enables experimental network filesystems. 9P (Plan 9) client for Linux discussed - would allow implementing filesystems as user-space servers. Projects: v9fs, Ron Minnich's work.

- [[https://lwn.net/Articles/69419/][snprintf() confusion]: C99 snprintf() returns length string "would be" if all fit (not actual length written), causing bugs in kernel. Many kernel coders expected the latter. Juergen Quade's patch adds scnprintf() and vscnprintf() that return actual length written. Plan: migrate kernel to new functions. Problem: many places do "len += snprintf(buf+len, buflen-len, ...)" - when buflen-len goes negative (size_t underflow), becomes huge value, inviting buffer overrun. Patch fixes 200+ calls.

- [[https://lwn.net/Articles/69523/][Trimming down sysfs]: sysfs uses VFS dentry/inode cache as backing store - one dentry+inode per file/directory. On large systems with hundreds/thousands of sysfs entries, consumes significant low memory. Maneesh Soni's patch: sysfs_dirent structures replace VFS caching for attribute files (70% of entries). Saves 145MB low memory on massive test system; active dentries drop from 60K to 9K. No changes to kobject structure. Getting closer to inclusion.

- [[https://lwn.net/Articles/70680/][Safe sysfs support]: Frame buffer sysfs support patch (merged in 2.6.3-rc1) had module unload oops bug. Problem: lifecycle rules - sysfs files can persist after device/module removed. fb_info structures freed twice or static ones freed.
