- [[https://lwn.net/Articles/339314/][Stallman warns about C# and Mono dependence]]: Jake Edge covers RMS's warning that relying on Mono or C# applications is a "serious danger" due to Microsoft software patent risk — triggered by Debian adding Mono-based Tomboy to a GNOME meta-package. - [[https://lwn.net/Articles/339337/][Soft updates, hard problems]]: Valerie Aurora explains why soft updates — the BSD FFS crash-consistency technique — is rarely implemented elsewhere despite impressive benchmarks (within 5% of in-memory FS performance vs. 20–30% slower for synchronous FFS). Soft updates avoids journaling by: (1) carefully ordering writes so only "allocated-but-free" inconsistencies can occur at crash time, then (2) running fsck in the background on a file system snapshot after reboot to reclaim leaked resources. The three ordering rules: never point to a structure before it's initialized; never re-use a resource before nullifying all pointers to it; never reset an old pointer before the new one is set. Cyclical dependencies (block A must precede B and B must precede A) are broken by lock/roll-back/write/roll-forward on each block write. The implementation requires 12 distinct dependency-tracking structures (bmsafemap, inodedep, allocdirect, indirdep, allocindir, pagedep, mkdir, dirrem, freefrag, freeblks, freefile) each manually designed per file system operation. The fatal flaw: every new FS feature or format change requires deriving all new ordering rules and custom structures from scratch — there is no general transaction abstraction. This tight coupling causes user-visible bugs (rmdir's ".." entry can delay parent link-count decrement by up to 2 minutes, breaking "rm -rf"). Journaling and copy-on-write provide a simpler open/add-block/close transaction interface requiring no special knowledge. Aurora concludes soft updates is "an evolutionary dead end."
- [[https://lwn.net/Articles/339349/][VA API slowly -- but surely -- making progress]]: Nathan Willis surveys the Video Acceleration API (VA API), launched December 2006 by Intel's Jonathan Bian as a modern replacement for XvMC, which only supported MPEG-2 motion compensation. VA API adds VC-1, MPEG-4 ASP/H.263, MPEG-4 AVC/H.264, iDCT, VLD (slice-level acceleration), deblocking, IZZ, color space conversion, scaling, and (since spec 0.30, March 2009) hardware video *encoding* (H.263, H.264, MPEG-4). Only Intel's closed-source psb-video driver for the Poulsbo/GMA500 chipset natively implements VA API; NVIDIA uses VDPAU and ATI uses XvBA, both proprietary and non-encoding. Gwenole Beauchesne (Splitted-Desktop Systems) wrote open-source bridge libraries (vdpau-video, xvba-video) allowing VA API calls to dispatch to VDPAU or XvBA backends, plus patched libVA and patched MPlayer/FFmpeg. VLC added VA API in its development branch. CPU utilization results: 66–98% unaccelerated vs. as low as 0.6% with VA API acceleration. S3 Graphics added a VA API driver for Chrome 400/500 in Feb 2009. Open source Nouveau and RadeonHD drivers are not pursuing hardware video acceleration. Future plans include G45 chipset support and rendering directly to OpenGL textures (for Clutter integration). The article notes VA API is the only vendor-neutral open hardware acceleration standard, but adoption remains limited by closed-driver dependency.
- [[https://lwn.net/Articles/340080/][Transcendent memory]]: Jonathan Corbet covers Dan Magenheimer's "transcendent memory" concept — a class of RAM of unknown and dynamically variable size, addressable only indirectly, configurable as persistent or ephemeral (may disappear without warning), and synchronously accessible. Implemented as page pools: tmem_new_pool() creates a pool identified by 128-bit UUID with flags for ephemeral bit, shared bit, page order, and version; tmem_put_page() / tmem_get_page() copy pages in/out by (pool_id, object_id, page_id) key tuple. Either call may fail — callers must have a fallback. Ephemeral tmem_get_page() may fail even after a successful put (hypervisor reclaimed the page). Two use cases: (1) precache — page cache stores about-to-be-evicted clean pages into an ephemeral pool, retrieving them later to avoid disk I/O; (2) preswap — swap writes to persistent pool instead of disk, saving I/O at both swap-out and swap-in. The hypervisor manages pool sizing, can coalesce duplicate pages (KSM-style), and can prioritize pools by guest activity. Key security concern: shared pools identified only by UUID allow a compromised guest to access another guest's data if it obtains the UUID; Dan dropped shared pool support in v2 pending a proper security model. Needs performance benchmarks before mainline inclusion.
- [[https://lwn.net/Articles/340400/][A lockless ring-buffer]]: Jake Edge covers Steven Rostedt's proposed lockless ring-buffer algorithm for the kernel's common tracing infrastructure (targeting 2.6.32), replacing the lock-heavy implementation merged in 2.6.28. The buffer is a circular doubly-linked list of pages with head, tail, and commit pointers — tail marks where new writes go, commit marks the latest completed write (needed because interrupt-nested writers can interleave). A separate reader page is detached from the list to prevent writer corruption. The lockless trick: pointers are 4-byte aligned, freeing 2 low bits for flags — HEADER (points to current head page) and UPDATE (page is volatile, being claimed by a writer). Readers swap the reader page for the head page using cmpxchg() requiring the HEADER flag; writers transitioning to a new tail page check for HEADER and set it to UPDATE, causing any concurrent reader cmpxchg() to fail and retry. Two modes: overwrite ("flight recorder") where new events overwrite old, and producer/consumer where writes to a full buffer fail. Edge case: if nested writes fill the buffer such that tail reaches the commit page on the reader page, writers must detect this and drop writes rather than push head forward (which would corrupt the commit pointer's path back into the ring). The algorithm depends on disabling preemption, ruling out direct use in user-space tracing. Mathieu Desnoyers (LTTng) notes complexity is "near that of RCU" and lacks formal proof; he plans an LTTng-based alternative. Subject to a Red Hat patent application by Rostedt, with a free-software use commitment.
- [[https://lwn.net/Articles/341110/][Communicating requirements to kernel developers]]: Jonathan Corbet covers a 2009 kernel summit pre-discussion thread (started by Matthew Garrett after the Gran Canaria Desktop Summit) on why userspace developers struggle to get requirements into the kernel. - [[https://lwn.net/Articles/341352/][A new way to truncate() files]]: Goldwyn Rodrigues covers Nick Piggin's patch series reworking the kernel's truncate() call sequence. The old sequence: do_sys_truncate() → do_truncate() → notify_change() → inode_setattr() → vmtruncate() (changes i_size first) → filesystem truncate() — problem: i_size is already changed before the filesystem is called, so the filesystem cannot fail the operation, doesn't know the previous i_size, and generic error-path block deallocation had no clean way to return errors. New sequence introduces: inode_newsize_ok(inode, offset) — centralises size limit and swap-file checks currently duplicated across filesystems; truncate_pagecache(inode, old, new) — unmaps pages beyond new size and should be called before filesystem releases disk blocks so writepage() is never called on deallocated blocks; simple_setsize() — replaces vmtruncate() in inode_setattr() when ATTR_SIZE is passed. Filesystems wanting the new behaviour set new_truncate=1 in inode_operations (admitted "nasty hack" for the transition period) and implement their own setsize() that validates size, truncates page cache, and frees disk blocks — clearing ATTR_SIZE before calling generic inode_setattr() to avoid double processing. vmtruncate() is deprecated. ext2 is the demonstration conversion (ext2_setsize()). The patchset is motivated by closing a race in page_mkwrite() where a truncate-beyond-current-size could leave PTEs pointing at uninitialized blocks.
- [[https://lwn.net/Articles/341244/][Is pre-linking worth it?]]: Jake Edge examines prelink, which pre-computes dynamic linker relocations and stores results in ELF sections to speed program startup — but Fedora Rawhide bugs have made its value questionable. The benefit: avoids most relocation cost at load time; pages that don't change for relocations stay shared via COW between processes (Ulrich Drepper's key motivation). Jakob Jelinek posted OpenOffice Writer numbers showing 10x relocation time difference. The cost: libraries at fixed addresses for 2 weeks weaken ASLR (address space layout randomization) — on 64-bit this is less concerning; binary integrity verification is harder (prelink modifies binaries); prelink must re-run after every library update; VDSO overlap on 32-bit forces some relocation anyway. Moblin (Arjan van de Ven) measured 15-20% disk read reduction during boot from avoiding page-fault loads of relocation data. Matthew Miller (Fedora) questioned whether fragility and forensic difficulty justify the gains for long-running servers. KDE's nix noted C++ libraries with thousands of long symbols (avg 31.9 chars) and hidden-visibility problems make relocation expensive even with GCC's .gnu.hash — prelink is "not small" for KDE-scale apps. The -Bdirect linking proposal (which would resolve symbols at compile time) was shot down by Drepper ("we have prelinking").
- [[https://lwn.net/Articles/342330/][Fun with NULL pointers, part 1]]: Jonathan Corbet dissects Brad Spengler's 2.6.30 local kernel exploit. The chain: Herbert Xu's TUN driver patch added a sock pointer dereference before a NULL check in tun_chr_poll(); the NULL tun pointer was reachable because Herbert Xu's accounting patch set it to NULL on a specific ioctl; the zero page (normally unmapped to catch NULL derefs) was mappable because CONFIG_SECURITY modules (e.g. SELinux) override mmap_min_addr checks at compile time, and Red Hat's default SELinux policy allows mapping page zero; the exploit uses personality(SVR4) with a setuid pulseaudio to map the zero page; GCC optimizes away the NULL check since the pointer was already dereferenced; the writable NULL lets test_and_set_bit(SOCK_ASYNC_NOSPACE) flip a single bit in memory — pointing into the TUN file_operations mmap pointer (normally NULL) and setting it to 1; calling mmap() on the TUN fd jumps to address 1 in the zero page, which contains exploit shellcode, achieving arbitrary kernel code execution. Multiple failures required: security modules overriding mmap_min_addr; SELinux allowing zero-page maps; pulseaudio setuid-root with module-loading; GCC NULL-check elimination; TUN mmap pointer being flippable. taviso and Julien Tinnes had independently discovered the personality() mmap_min_addr bypass. Fix: move the dereference after the NULL check; disable page-zero mapping by default.

- [[https://lwn.net/Articles/343608/][A new GCC runtime library license snag?]]: Jonathan Corbet covers a licensing problem Florian Weimer raised with GCC 4.4's runtime library (libgcc). GCC 4.4 moved to GPLv3, with an exception allowing proprietary use — but this creates incompatibility with GPLv2-only programs like git and udev. The issue: GPLv2's "system library" exception says linking a system library with a GPLv2 program doesn't create a derived work — but it contains an "unless that component itself accompanies the executable" clause. Since distributors ship GCC (the "component") alongside programs, the exception doesn't apply to the GCC runtime library, making distribution of GPLv2-only programs with libgcc technically a GPLv2 violation. The perverse result: GPLv2-only projects can't be distributed with GCC 4.4. The FSF's runtime exception tries to solve this but can't bind non-FSF copyright holders. Debian (which triggered the discovery) faces a dilemma. Potential fixes: FSF issue a statement blessing the combination (not binding on others); relicense libgcc to GPLv2+ (foom's suggestion); or await a fork. Corbet notes the FSF has been unresponsive to Weimer's queries since April 2009.

- [[https://lwn.net/Articles/344117/][A desktop "secrets" API]]: Jake Edge covers a freedesktop.org initiative by Michael Leupold (KDE Wallet) and Stef Walter (GNOME Keyring) to create a unified Secrets API across desktop environments. Users run Keyring or Wallet as a login-session daemon; applications use the D-Bus-based API without caring which backend provides it. Secrets are opaque byte arrays with lookup attributes (key-value pairs), labels, properties, and can be grouped into collections. Items/collections can be locked, requiring user password to unlock. The API allows optional encryption in transit, but docs explicitly warn it won't withstand man-in-the-middle or active attacks — intended to prevent plaintext in swap/memory caches, not to secure the D-Bus session. Java/C# /Python apps can't prevent their own memory paging, making encrypted transit of limited value for them. Higher-level topics under discussion: browser password identification for cross-browser migration, secret never leaving the daemon (for smartcards), and "smart user accounts" where a privileged daemon acts on behalf of the user without revealing secrets. Mozilla hasn't engaged; Firefox/Thunderbird integration lags. Plan 9's secstore/factotum cited as a more sophisticated prior art.
